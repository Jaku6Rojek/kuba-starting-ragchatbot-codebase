Course Materials RAG · request lifecycle

How one question moves through the system

A single POST /api/query doesn't just retrieve and answer. Claude is handed a search tool and decides whether to use it — so the interesting work happens in an agentic loop between the model and the vector store.

Plumbing & orchestration Model — Claude generates / decides Retrieval — vector search
1

frontend/script.js

Browser sends the question

POST /api/query with { query, session_id }. The session id is null on the first turn.

2

backend/app.py · FastAPI

Endpoint creates a session if needed

Mints a session via SessionManager when none was sent, then calls rag_system.query().

3

backend/rag_system.py · orchestrator

Assemble history + tools, call the generator

Pulls prior turns from SessionManager and the tool schema from ToolManager, then hands everything to AIGenerator.generate_response().

4
agentic loop · ai_generator.py

Claude call #1 — with tools

The model decides: do I need to search?

System prompt routes the question. General knowledge is answered from the model's own weights; course-specific questions trigger a tool call.

stop_reason ≠ tool_use → answer directly stop_reason = tool_use → search ↓

search_tools.py → vector_store.py

CourseSearchTool queries ChromaDB

Runs the semantic search (detail below), formats the hits for the model, and stashes UI source labels in self.last_sources.

↑ results appended to the conversation, then Claude is called again without tools to write the final answer

5

backend/rag_system.py → app.py

Collect sources, save history, respond

Sources are read via get_last_sources() then reset; the exchange is written back to the session. The browser receives { answer, sources, session_id }.

Note — sources travel out-of-band. They never pass through the model's text. The tool records them as a side effect, and the orchestrator picks them up after generation finishes, then clears them for the next query.
▾

vector_store.py · ChromaDB · two collections

Why retrieval is a two-step search

A fuzzy course name is first resolved to its exact title, then used to filter the content. The course title is the unique ID across the whole system.

course_catalog

One entry per course. Document = title; metadata holds instructor, links, and lessons (as a lessons_json string).

course_content

Chunked lesson text — the embeddings actually searched for relevant passages.

"MCP" → resolve against catalog → exact title → filter & search content
